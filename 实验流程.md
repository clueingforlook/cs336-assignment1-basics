**(a) What Unicode character does `chr(0)` return?**

> **Deliverable:** A one-sentence response.

**Answer:**
It returns the Null character (often represented as `\x00`).
*(返回空字符 \x00)*

---

**(b) How does this character’s string representation (`__repr__()`) differ from its printed representation?**

> **Deliverable:** A one-sentence response.

**Answer:**
The string representation (`__repr__`) shows the escape sequence `'\x00'` to explicitly indicate its presence, while the printed representation renders nothing (it is invisible/non-printing).

* **交互示例：**
  * 直接输入 `chr(0)` -> 显示 `'\x00'`
  * 输入 `print(chr(0))` -> 显示为空白（看不到任何东西）

## 2.2 Unicode Encodings

Problem (unicode2): Unicode Encodings (3 points)
(a) What are some reasons to prefer training our tokenizer on UTF-8 encoded bytes, rather than
UTF-16 or UTF-32? It may be helpful to compare the output of these encodings for various
input strings.
Deliverable: A one-to-two sentence response.

> * 空间效率：互联网上的文本（以及代码）大部分是英文。UTF-8 对英文（ASCII）只需要 1 个字节，而 UTF-32 无论什么字符都要 4 个字节，UTF-16 至少要 2 个字节。

> * 稀疏性：如果你用 UTF-32，会有大量的 0x00 字节（空字节）填充，这对于模型训练来说是无效信息的浪费。

(b) Consider the following (incorrect) function, which is intended to decode a UTF-8 byte string into
a Unicode string. Why is this function incorrect? Provide an example of an input byte string
that yields incorrect results.
def decode_utf8_bytes_to_str_wrong(bytestring: bytes):
return "".join([bytes([b]).decode("utf-8") for b in bytestring])

```python
>>> decode_utf8_bytes_to_str_wrong("hello".encode("utf-8"))
'hello'
```

Deliverable: An example input byte string for which decode_utf8_bytes_to_str_wrong produces incorrect output, with a one-sentence explanation of why the function is incorrect.

> UTF-8 是变长编码。
> “牛”字在 UTF-8 里由 3 个字节组成（例如 [227, 137, 155]）。
> 这个错误的函数试图把 227 单独拿出来解码，把 137 单独拿出来解码。
> 但是在 UTF-8 规则里，这些字节必须合在一起才有意义，单独拆开就是无效的乱码。

(c) Give a two byte sequence that does not decode to any Unicode character(s).
Deliverable: An example, with a one-sentence explanation.
合法编码：

| 字节数 | 格式 (二进制)                       | 含义                                                                           |
| :----- | :---------------------------------- | :----------------------------------------------------------------------------- |
| 1 字节 | 0xxxxxxx                            | 单字节字符 (ASCII)。看到 0 开头，它就是独立的。                                |
| 2 字节 | 110xxxxx 10xxxxxx                   | 110 开头是火车头，告诉计算机“我要占2个字节”。后面必须跟 1 个 10 开头的字节。 |
| 3 字节 | 1110xxxx 10xxxxxx 10xxxxxx          | 1110 开头是火车头，后面必须跟 2 个 10 开头的字节。                             |
| 4 字节 | 11110xxx 10xxxxxx 10xxxxxx 10xxxxxx | 11110 开头是火车头，后面必须跟 3 个 10 开头的字节。                            |

> 非法编码：0xFF

## 2.8 Section 7 实验命令与 JSON 配置用法

下面这一节补充当前仓库里已经可用的一套实验运行方式，目标是解决两个问题：

1. PowerShell 多行命令不方便直接复制粘贴
2. Section 7 的参数比较多，适合保存成配置文件复用

### 2.8.1 为什么可以直接复用已经训练好的 tokenizer

如果你已经用 `profile_bpe.py` 训练出了：

- `tinystories_tokenizer_vocab.json`
- `tinystories_tokenizer_merges.json`

那么这两个文件可以直接给 `prepare_token_bin.py` 使用。

原因：

- `profile_bpe.py` 用的是 `Tokenizer.save(...)`
- `prepare_token_bin.py` 用的是 `Tokenizer.from_files(...)`
- 两边使用的是同一套 `cs336_basics.bpe.Tokenizer` 文件格式

因此不需要重新训练 tokenizer，也不需要改训练代码。

### 2.8.2 一行版命令

#### 1. 把 TinyStories 训练集编码成 `.bin`

```powershell
uv run .\prepare_token_bin.py --input_txt .\data\TinyStoriesV2-GPT4-train.txt --output_bin .\data\tinystories_train.bin --vocab_json .\tinystories_tokenizer_vocab.json --merges_json .\tinystories_tokenizer_merges.json
```

Windows PowerShell:

上面这一条就是 Windows PowerShell 命令。

Linux/macOS shell:

```bash
uv run ./prepare_token_bin.py --input_txt ./data/TinyStoriesV2-GPT4-train.txt --output_bin ./data/tinystories_train.bin --vocab_json ./tinystories_tokenizer_vocab.json --merges_json ./tinystories_tokenizer_merges.json
```

作用：

- 读取 TinyStories 训练文本
- 用已经训练好的 tokenizer 做编码
- 生成一维 token 流文件 `data/tinystories_train.bin`

预期输出：

- 周期性打印 `encoded lines=... tokens=...`
- 结束时打印 `Finished. output=... total_tokens=...`
- 同目录生成 `tinystories_train.bin.meta.json`

#### 2. 把 TinyStories 验证集编码成 `.bin`

```powershell
uv run .\prepare_token_bin.py --input_txt .\data\TinyStoriesV2-GPT4-valid.txt --output_bin .\data\tinystories_valid.bin --vocab_json .\tinystories_tokenizer_vocab.json --merges_json .\tinystories_tokenizer_merges.json
```

作用：

- 读取 TinyStories 验证文本
- 生成验证集 token 流 `data/tinystories_valid.bin`

预期输出：

- 周期性打印编码进度
- 结束后生成 `tinystories_valid.bin` 和 `tinystories_valid.bin.meta.json`

#### 3. 启动 Section 7 训练

```powershell
uv run .\run_section7_training.py --experiment_name tinystories_baseline --train_data_path .\data\tinystories_train.bin --val_data_path .\data\tinystories_valid.bin --vocab_size 10000 --context_length 256 --d_model 512 --num_layers 4 --num_heads 16 --batch_size 32 --max_iters 5000 --eval_interval 100 --save_interval 500 --device cuda:0
```

作用：

- 从训练集 `.bin` 随机采样 batch
- 周期性在验证集上计算 `val_loss`
- 记录 step、wallclock、learning rate、loss
- 保存 checkpoint 和最终 summary

预期输出：

- 控制台打印 `step ... | train_loss ... | val_loss ... | lr ...`
- 周期性打印 `Saved checkpoint to ...`
- 最终在 `runs/tinystories_baseline_<timestamp>/` 下生成：
  - `config.json`
  - `metrics.csv`
  - `summary.json`
  - `checkpoints/`

### 2.8.3 JSON 配置文件方式

为了避免每次手写很长的命令，可以用当前仓库里的：

- `run_with_json.ps1`
- `prepare_train.json`
- `prepare_valid.json`
- `section7_train.json`

#### 1. `run_with_json.ps1` 的作用

这个脚本是一个 PowerShell 包装器，不会修改 Python 脚本本身的逻辑。

它做的事情只有三步：

1. 读取 JSON 文件
2. 把 JSON 的键值对展开成命令行参数
3. 调用 `uv run <script> --key value ...`

例如：

```json
{
  "batch_size": 32,
  "device": "cuda:0"
}
```

会被展开成：

```powershell
--batch_size 32 --device cuda:0
```

所以本质上，Python 代码依然是普通的 `argparse`；只是 PowerShell 帮你自动拼好了命令。

#### 2. 编码训练集

```powershell
.\run_with_json.ps1 -Script .\prepare_token_bin.py -Config .\prepare_train.json
```

作用：

- 读取 `prepare_train.json`
- 自动展开为 `prepare_token_bin.py` 的参数
- 生成 `data/tinystories_train.bin`

#### 3. 编码验证集

```powershell
.\run_with_json.ps1 -Script .\prepare_token_bin.py -Config .\prepare_valid.json
```

作用：

- 读取 `prepare_valid.json`
- 生成 `data/tinystories_valid.bin`

#### 4. 启动训练

```powershell
.\run_with_json.ps1 -Script .\run_section7_training.py -Config .\section7_train.json
```

作用：

- 读取 `section7_train.json`
- 自动拼接训练参数
- 启动 Section 7 实验训练

### 2.8.4 当前三个 JSON 文件分别控制什么

#### `prepare_train.json`

控制训练集编码参数：

- 输入文本路径
- 输出 `.bin` 路径
- tokenizer 的 vocab / merges 文件
- token 存储 dtype
- `special_tokens`

#### `prepare_valid.json`

控制验证集编码参数：

- 输入验证文本
- 输出验证 `.bin`
- 其余 tokenizer 参数应与训练集保持一致

#### `section7_train.json`

控制模型训练参数：

- 实验名 `experiment_name`
- 训练集和验证集 token bin 路径
- 模型大小
- batch size
- 最大训练步数
- 验证频率
- checkpoint 频率
- 设备类型

### 2.8.5 使用时的注意事项

- 所有命令都应在项目根目录执行
- 如果更换 tokenizer，就必须重新生成 `.bin`
- `run_section7_training.py` 需要 train/val 两个独立的 token 文件
- 如果本机没有 CUDA，把 `section7_train.json` 里的 `"device": "cuda:0"` 改成 `"cpu"`
- 如果显存不够，优先减小 `batch_size`
